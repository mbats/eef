Introduction

In this document, you will find all the necessary information used to start contributing to EEF. From the configuration of your development environment to some details on the various tools and settings to use, everything will be explained here.

The goal of the project is to have Properties view and wizards perfectly integrated to Eclipse Sirius and Obeo Designer Team (client / server and asynchronous context). EEF should also provide implementations of Dialogs, EObjects selectors and files selectors easily reusable in other contexts. The user interface should support an asynchronous context where fields could be initialized without any value in a read only mode and updated later on.

The less code, the better. EEF should be easily maintainable and properly structured. Its concepts should be easily extended.

The user interface should be dynamic and parameterized in a similar fashion as Eclipse Sirius.

The project should have a modular structure with APIs structured in a similar way. The core components should not depend on any user interface toolkit (i.e SWT), the components depending on a specific toolkit should be isolated and even the components using EMF edit should not depend on a user interface toolkit.

Development Environment

  • Eclipse Mars SDK Release
  • Checkstyle 6.5
  • Target Platform DSL
  • Jekyll 2.5.x

The target platform of the project should support the same platforms as Sirius.

API Documentation

  • All the APIs need to be documented with, at least, the following information
  • Interface of the API?
  • What can of instances are required? (EPackage, EClass, etc) and why?
  • Lifecycle of the API? (creation, deletion)

The documentation of the API should also contain a scenario indicating which instances are involved and when.

Development Process

  • Continuous integration
  • Code review with Gerrit
  • Static analysis
  • Unit tests
  • Respecting all the best practices of Eclipse projects development (no warnings, API restrictions)

Dependencies

The core components of EEF should work with the minimum amount of dependencies possible.

  • EMF Common
  • EMF Ecore
  • EMF Edit
  • Guava

Guava should not appear in any API of the EEF project!

Technical Guidelines

EEF 1.x and 2.x should be able to work side by side, as such both projects should not use the same bundle name or package name. The core bundles of EEF 1.x uses the namespace org.eclipse.emf.eef. The bundle of EEF 2.x could thus use the namespace org.eclipse.eef to prevent any possible confusion. The URIs of the meta-models of EEF 2.x should also use a different namespace (http://www.eclipse.org/eef/...).

Some components used by EMF project should either not be used directly by EEF or used in a very specific manner. Some of those could be used by EEF but EEF should not instance them by itself.

  • UsageCrossReference & ECrossReferenceAdapter
  • eAllContents()
  • EcoreUtil
  • eResource()
  • ResourceSet
  • EPackageRegistry - We should only read the EPackages available and it should not automatically load the EPackage directly
  • add() & addUnique() - We should only use addUnique() in the create of the user interface model
  • EMF Edit - Everything displayed to the end user should use the label provider from EMF Edit
  • AdapterFactories and the XXXProvider should be properly disposed
  • Adapter & ContentAdapter
  • Do not install a ChangeRecorder on the ResourceSet

We should also prevent leaks by not having references to

  • ResourceSet
  • Resource
  • EObject (small number of dependencies to instances of EObject with a short lifecycle)
